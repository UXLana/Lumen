1. What This Is

This document is an accessibility audit of the data tables used across Metrc. Every data table in Metrc uses a component called "Kendo UI Grid" (made by Telerik) configured with the ARIA "treegrid" pattern. This pattern allows expandable rows — for example, clicking a plant row to see its history.

The treegrid pattern has strict accessibility requirements defined by the WAI-ARIA 1.2 specification. When implemented incorrectly, screen reader users cannot navigate or understand the table structure, and keyboard-only users cannot expand/collapse rows.

We audited 3 pages (Plants, Packages, Licensed Transfers) and found that all tables share the same 15 ARIA violations because they all use the same underlying widget. This means a single centralized fix can potentially remediate every table site-wide.

2. Background

The State of Illinois requires WCAG 2.1 AA compliance for Metrc under the Rehabilitation Act (Section 508). A state-contracted accessibility tester has identified treegrid violations as a critical blocker. The state tester has indicated that the treegrid pattern is "all or nothing" — partial implementation does not satisfy compliance.

Key deadlines:

  • March 31, 2026 — Remediation demonstrated in test environment
  • June 30, 2026 — Final sign-off

3. Executive Summary

We identified 15 ARIA violations in Metrc's treegrid implementation and confirmed they are identical across all 3 audited pages. Every page uses the same Kendo UI Grid widget (v2019.3.917) with the same configuration, producing the same accessibility failures.

Severity breakdown:

  • 6 Critical (P0) — Complete barriers for assistive technology users
  • 5 Serious (P1) — Major navigation or comprehension failures
  • 4 Moderate (P2) — Degraded experience, incomplete information

Pages confirmed:

  • Plants — 15 treegrids, all 15 violations
  • Packages — 4 treegrids + 1 plain grid, all 15 violations
  • Licensed Transfers — 5 treegrids + nested grids in detail rows, all 15 violations

4. How We Tested

Testing was performed on the QA environment (qa.metrc.com) using the Grower 01 license (4a-X0001). We inspected the DOM structure using browser developer tools to verify ARIA attributes, roles, and keyboard accessibility against the WAI-ARIA 1.2 Treegrid Pattern specification.

For each page, we checked:

  • Kendo UI version and configuration (navigatable, selectable, detailTemplate)
  • ARIA roles on table, rows, cells, and headers
  • Required treegrid attributes (aria-level, aria-expanded, aria-rowindex, etc.)
  • Keyboard operability of expand/collapse controls
  • Tab and tablist implementation
  • Pagination control labeling

6. What Is the POC

The Proof of Concept (POC) is a self-contained JavaScript file that fixes all 15 ARIA violations at runtime. When a developer pastes it into the browser console on a QA page, it modifies the DOM to add the missing ARIA attributes, fix keyboard accessibility, and correct the treegrid structure.

The POC is for validation only — it proves the fixes work and gives screen reader users a correct experience. It does not persist after a page refresh. The purpose is to let engineering confirm the approach before building a production implementation.

8. Step-by-Step Testing Guide

Part A: Open the Browser Console

  1. Navigate to a Metrc QA page with data tables (e.g., /industry/{license}/plants)
  2. Open browser developer tools:
    • Mac: Right-click anywhere on the page → Inspect, then click the Console tab. Or press Cmd + Option + J to open the console directly.
    • Windows: Press F12, then click the Console tab.
  3. Click in the console input area at the bottom

Part B: Run the POC Script

  1. Copy the entire script from Section 7
  2. Paste it into the console
  3. Press Enter
  4. You should see a confirmation message: [Metrc A11y] Treegrid accessibility fixes applied to X grids

Part C: Verify with Screen Reader (VoiceOver on Mac)

  1. Press Cmd + F5 to enable VoiceOver
  2. Navigate to the data table using VO + arrow keys
  3. Verify VoiceOver announces:
    • The table label (e.g., "Mother Plants, treegrid")
    • Row count and column count
    • Row hierarchy (e.g., "level 1, row 1 of 13")
    • Expanded/collapsed state when toggling rows
  4. Navigate to the tabs and verify:
    • Tab names and selected state are announced
    • Pipe separators are not announced as tabs

Part D: Verify Keyboard Accessibility

  1. Click on the first data row in a table
  2. Press Tab to move to the expand/collapse arrow
  3. Press Enter or Space — the row should expand
  4. Press Enter or Space again — it should collapse
  5. Navigate to the tab bar and use Left/Right arrow keys between tabs
  6. Verify the page number input and rows-per-page dropdown are labeled (screen reader should announce "Current page number" and "Rows per page")

Part E: Inspect DOM Attributes

  1. Right-click a data table → Inspect
  2. Find the <table role="treegrid"> element and verify it has:
    • aria-label (e.g., "Mother Plants")
    • tabindex="0"
    • aria-rowcount and aria-colcount
  3. Find a <tr role="row"> data row and verify it has:
    • aria-level="1"
    • aria-setsize and aria-posinset
    • aria-rowindex
    • aria-expanded on the row (not on a cell)
  4. Expand a row and check the detail row has:
    • role="row" and aria-level="2"
    • Any nested grid has an aria-label

10. Production Recommendations

The POC validates the approach. For production, we recommend:

  1. Create a centralized Kendo Grid accessibility module that hooks into Kendo's dataBound, detailInit, and sort events.
  2. Apply the module globally so it covers all existing and future grids automatically.
  3. Enable navigatable: true in the Kendo Grid configuration for keyboard navigation support.
  4. Add the module to the build pipeline and include it on every page that uses Kendo Grid.
  5. Regression test with VoiceOver (Mac), NVDA (Windows), and keyboard-only navigation.
